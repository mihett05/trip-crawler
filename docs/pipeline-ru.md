# Описание пайплайна сбора данных

## 1. Общая архитектура

Система состоит из трёх независимых компонентов, общающихся через очередь NATS JetStream:

```
┌─────────────┐     NATS JetStream      ┌─────────────┐
│  Scheduler  │ ──── STATIONS.load ───► │             │
│  (сервис)   │ ──── TICKETS.parse ───► │   Worker    │
└─────────────┘                         │  (воркер)   │
       │                                └──────┬──────┘
       │ читает                                │ пишет
       ▼                                       ▼
┌─────────────────────────────────────────────────────┐
│                      Dgraph                         │
│        (граф: города → станции → поездки)           │
└─────────────────────────────────────────────────────┘
       ▲
       │ читает
┌─────────────┐
│  HTTP API   │  ◄── запросы пользователей
│  (сервис)   │
└─────────────┘
```

## 2. Поток 1 — Загрузка станций (раз в сутки, 02:00)

**Цель:** обновить граф городов, станций и связей между ними.

### Шаги

**1. Scheduler публикует задачу**

Каждый день в 02:00 планировщик отправляет одно сообщение в очередь `STATIONS.load`:

```json
{ "requested_at": "2026-03-28T02:00:00Z", "top_n": 100 }
```

**2. Worker получает задачу — `StationLoadProcessor`**

```
Скачать страницу Википедии «Города России»
    └─► Распарсить HTML-таблицу (goquery)
        └─► Получить список CityData {Name, Region, Population}
            └─► Отсортировать по населению по убыванию
                └─► Взять топ-N городов (MVP: 100)
```

**3. Группировка городов по префиксу и запрос к RZD Suggest API**

Вместо отдельного запроса на каждый город, список городов разбивается на префиксные группы с помощью `utils.BuildPrefixGroups` (префикс 3 символа, максимум 20 городов в группе). На каждую группу делается **один** вызов `SuggestCity(prefix)`.

```
BuildPrefixGroups(cities, prefixLen=3, maxGroupSize=20)
    └─► map[prefix][]CityData
        │  Например: "Мос" → [Москва], "Сан" → [Санкт-Петербург], ...
        │
        └─► Для каждой группы:
                Throttler.Wait(ctx)              ← ограничение частоты
                rzd.SuggestCity(prefix)
                    └─► resp.City[]  — RZD-узлы городов { NodeID, Name }
                    └─► resp.Train[] — станции { CityID, ExpressCode, TransportType }
                        │
                        ├─► Индекс: normalised(Name) → NodeID   (из resp.City)
                        └─► Индекс: NodeID → []Station           (из resp.Train, rail only)
                            │
                            └─► Для каждого города в группе:
                                    Name → NodeID → []Station
```

**Эффект:** для топ-100 городов вместо 100 запросов делается ~5–15 (по числу уникальных 3-символьных префиксов). Каждый вызов SuggestCity проходит через **Throttler** — см. раздел 5.

**4. Upsert в Dgraph**

Для каждого города сохраняется граф:

```
City { Name, Population }
  └── has_station ──► Station { ExternalID, Name, TransportType }
```

## 3. Поток 2 — Загрузка билетов (раз в сутки, 03:00)

**Цель:** для каждой актуальной пары станций и каждой нужной даты запросить цены на поезда.

### Шаги

**1. Scheduler читает граф и вычисляет задачи**

```
Получить топ-100 городов из Dgraph (GetTopConnections)
    └─► Для каждой связи (origin → destination):
            Для каждой даты departure в диапазоне [сегодня+1 .. сегодня+90]:
                Проверить: shouldParse(connection, today, date)?
                    └─► Если да → опубликовать TicketParseTask в TICKETS.parse
        └─► После обхода всех дат → MarkParsed(connection, today)
```

Публикация ограничена до `SCHEDULER_PUBLISH_RATE` сообщений/сек (по умолчанию 500).

**2. Алгоритм shouldParse — матрица частоты**

Решение «парсить или пропустить» основано на трёх факторах:

| Параметр | Описание |
|---|---|
| **Тир города** | A (>1М чел.), B (500К–1М), C (100К–500К), D (<100К) |
| **Горизонт даты** | Ближние (1–14 дн.), средние (15–45 дн.), дальние (46–90 дн.) |
| **Спящий режим** | Если связь не использовалась в маршрутах >30 дней → интервал 21 день |

Матрица интервалов (в днях между парсингами):

| Маршрут | 1–14 дн. | 15–45 дн. | 46–90 дн. |
|---------|----------|-----------|-----------|
| A ↔ A | 1 | 2 | 4 |
| A ↔ B | 1 | 3 | 7 |
| A ↔ C | 2 | 5 | 14 |
| A ↔ D | 3 | 7 | 21 |
| B ↔ B | 1 | 4 | 10 |
| B ↔ C | 2 | 7 | 14 |
| B ↔ D | 3 | 10 | 21 |
| C ↔ C | 3 | 10 | 21 |
| C ↔ D | 5 | 14 | 30 |
| D ↔ D | 7 | 21 | 60 |

Пример: маршрут Москва (A) ↔ Сыктывкар (B) на дату через 10 дней → интервал 1 день → парсим каждый день. Тот же маршрут на дату через 60 дней → интервал 7 дней → парсим раз в неделю.

**3. Worker получает задачу — `TicketParseProcessor`**

```
Throttler.Wait(ctx)                  ← ограничение частоты
    └─► ProxyManager.GetClient()     ← получить HTTP-клиент с прокси
        └─► rzd.ParseTrains(origin, dest, date)
            ├─► Успех (200 OK)
            │       └─► Конвертировать Train[] → []Train{ExternalID, Price, ...}
            │           └─► TripRepository.SaveTrains()
            │               └─► msg.Ack()
            │
            ├─► Бан прокси (429 / 403)
            │       └─► ProxyManager.Ban(httpClient)
            │           └─► msg.Nak(delay=1 мин)   ← задача вернётся другому воркеру
            │
            └─► Сетевая ошибка / таймаут
                    └─► msg.Nak(delay=30 сек)
```

**4. Upsert поездок в Dgraph**

```
Station ──── departs ────► Trip { ExternalID, Price, DepartureAt, ArrivalAt }
                                └── destination ──► Station
```

## 4. Жизненный цикл задачи в NATS

```
Scheduler                 NATS JetStream               Worker
────────────────────────────────────────────────────────────────
publish(msg)  ──────────►  [In Queue]
                           [In Progress] ◄── subscribe(msg)
                                              process...
                           [Acked / Done] ◄── msg.Ack()        ✓ успех
                           [In Queue, +30s] ◄─ msg.Nak(30s)   ✗ ошибка сети
                           [In Queue, +1m] ◄── msg.Nak(1min)  ✗ бан прокси
                           [Dropped] ◄──────── msg.Ack()       ✗ битый payload
```

Политика очереди — `WorkQueuePolicy`: каждое сообщение доставляется ровно одному воркеру. Максимальный возраст задачи в очереди — 24 часа; устаревшие задачи автоматически удаляются.

## 5. Ограничение частоты запросов к РЖД (Throttler)

Чтобы не получить бан, все HTTP-запросы к RZD проходят через общий `Throttler`:

```
Запрос 1  ──► пропустить (счётчик: 1/5)
Запрос 2  ──► пропустить (счётчик: 2/5)
Запрос 3  ──► пропустить (счётчик: 3/5)
Запрос 4  ──► пропустить (счётчик: 4/5)
Запрос 5  ──► СТОП → спать rand(5s, 15s) → сбросить счётчик → пропустить
Запрос 6  ──► пропустить (счётчик: 1/5)
...
```

Конфигурация:

| Переменная | По умолчанию | Описание |
|---|---|---|
| `THROTTLE_BURST_SIZE` | `5` | Сколько запросов в одном всплеске |
| `THROTTLE_SLEEP_MIN` | `5s` | Минимальная пауза между всплесками |
| `THROTTLE_SLEEP_MAX` | `15s` | Максимальная пауза между всплесками |

Throttler разделяется между всеми горутинами воркера (один экземпляр, мьютекс), поэтому лимит глобальный, а не на горутину.

## 6. Поиск маршрутов (HTTP API)

Пользователь не взаимодействует со сборщиком данных напрямую. Его запросы идут только в HTTP API сервиса:

```
GET /route?from=Москва&to=Сыктывкар&date=2026-04-10
    └─► DgraphItineraryBuilder.Build()
            └─► Обход графа: City → Station → Trip → Station → City
                └─► Найти оптимальный маршрут с пересадками
                    └─► Вернуть []RoutePoint { Name, StartAt, EndAt, Details }
```

Каждый раз, когда алгоритм использует связь между станциями, обновляется `LastUsedAt` на рёбрах графа. Именно это поле используется планировщиком для определения «спящих» маршрутов.

## 7. Схема данных в Dgraph

```
City
  ├── city.name        : string  (индекс term)
  ├── city.population  : int
  └── has_station ──►  Station
                         ├── station.name           : string
                         ├── station.transport_type : string  (= "rail")
                         ├── station.external_id    : string  (7-значный код РЖД)
                         └── departs ──►  Trip
                                           ├── trip.external_id   : string
                                           ├── trip.price         : float
                                           ├── trip.departure_at  : datetime
                                           ├── trip.arrival_at    : datetime
                                           └── destination ──►  Station
```

## 8. Конфигурация планировщика

| Переменная | По умолчанию | Описание |
|---|---|---|
| `SCHEDULER_TOP_N` | `100` | Топ-N городов по населению |
| `SCHEDULER_DAYS_AHEAD` | `90` | Горизонт дат для парсинга |
| `SCHEDULER_PUBLISH_RATE` | `500` | Макс. сообщений/сек в NATS |
| `SCHEDULER_STATION_HOUR` | `2` | Час запуска загрузки станций |
| `SCHEDULER_TICKET_HOUR` | `3` | Час запуска генерации задач |

## 9. Инфраструктура

| Компонент | Технология | Роль |
|---|---|---|
| Граф БД | Dgraph | Хранение городов, станций, поездок |
| Очередь | NATS JetStream | Распределение задач между воркерами |
| Сервис | Go + chi | HTTP API, планировщик |
| Воркер | Go | Парсеры, процессоры задач |
| Наблюдаемость | OpenTelemetry + Zap | Трейсы, метрики, логи |
